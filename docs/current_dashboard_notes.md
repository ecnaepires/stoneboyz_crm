## 1. Current Tech Stack

Frontend: Next.js App Router in `apps/web`, React server components plus server actions.

Backend: NestJS API in `apps/api`.

Database: PostgreSQL via SQL migrations in `db/migrations`, queried with `pg`.

Auth: Session auth plus Better Auth style routes in `apps/web/src/app/api/auth`, guarded in API by session/role guards.

Domain/shared types: `packages/domain`.

API client: OpenAPI-generated client in `packages/api-client`.

Calendar library: No external full-calendar library observed. Current scheduling UI is custom React/Next markup.

File storage: Storage service exists, currently used for slab image upload. Generic attachment records exist, but job file upload UI is not wired.

## 4. Current Job Creation Workflow

Current steps:

1. User opens `/projects`.
2. User clicks `+ New Project`.
3. User selects an existing customer.
4. User enters title and status, optionally description.
5. Server action posts to `/projects`.
6. User is redirected to `/projects/{id}`.

Does job creation require a job name? Yes, under the field label `Title *`.

Does job creation require an account? Partial. It requires `customerId`, but the UI says Customer, not Account.

Can a user create a new account during job creation? No.

Can a user select a job template during job creation? No in the UI. Partial in the backend/domain, because `CreateProjectInput` and `createProjectSchema` include optional `jobTemplateId`, and `/job-templates` can list templates.

Notes:

- The desired button says `Create Job`; current button says `+ New Project`.
- The desired first fields are Job Name, Account, and Job Template. Current fields are Customer, Title, Status, Description.
- No account search/typeahead exists in the create-job form; it loads up to 100 customers into a plain select.

Can account address and job address be edited separately? Partial.

- Phases and phase checklists exist.
- Quotes, quote drawing/measurements/pricing, orders, order payments, and pipeline stage logic exist.

### 18.3 What Partially Exists

### 18.8 Autoscheduling/Calendar Logic Needed

Autoscheduling should be server-owned.

Recommended first autoschedule behavior:

- Job template activity specs define order, default duration, dependency, and offset.
- When the user schedules the first main activity, the backend schedules dependent activities by applying offsets.
- Store generated activities/events with `autoschedule_state = autoscheduled`.
- If a user manually edits an autoscheduled activity, mark it `manual_override`.
- When an upstream activity changes, recompute only unlocked/autoscheduled dependents.
- For manually overridden dependents, either preserve the override or return a prompt payload asking whether to update overridden dependents.
- Calendar creation/update should use the same service as the activity editor.

Calendar behavior needed:

- Activities with scheduled date/time/duration appear on the calendar.
- Clicking calendar activity opens the same activity editor.
- Drag/drop rescheduling updates the activity/scheduled event and invokes autoschedule dependency logic.
- Calendar should display job, account, activity type, assignees, and status.

Open scheduling decisions:

- Business hours and non-working days.
- Whether weekends are allowed.
- Whether offsets are calendar days, business days, or shop working days.
- Conflict detection rules.
- Whether assignees/resources can be double-booked.
- Whether autoscheduling starts from Template only or from any first scheduled activity.

### 18.9 What Should Be Built First

Build the job-centered foundation first:

1. Align terminology and creation flow: Job Name, Account, Job Template.
2. Add account quick-create from job creation.
3. Persist selected job template on the job.
4. Generate job activity instances from the selected template in a not-scheduled state.
5. Update job detail to show real job activities, account address, and job address separately.
6. Add activity editor for status/date/time/duration/notes using current user-based assignees temporarily.
7. Add separate Assignee model and migrate UI from users to assignees.
8. Add autoscheduling based on template dependencies/offsets.
9. Wire calendar editing into the same activity update/autoschedule service.
10. Add job files and configurable forms after the core job/activity/calendar loop is stable.

## 20. Risks And Watchouts

- The UI currently mixes Customer/Account and Project/Job terminology. Renaming everything at once would be risky; UI-first terminology alignment is safer.
- The current project detail page displays the primary customer address as Job Address. This could hide real job-address differences.
- The event detail page links to an edit route that appears missing.
- Inline scheduled event notes appear to have been removed, but some UI still references `event.notes`.
- Existing scheduled events require at least one `assigneeUserId`, which conflicts with the desired separate assignee/resource model.
- Autoscheduling should not be implemented in the frontend; backend must own official schedule/dependency changes.
- Financial flows already exist as quotes/orders/payments. Avoid calling orders/invoices/payments by the wrong names until the desired accounting terminology is decided.
